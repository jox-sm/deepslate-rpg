# Project Context - Deepslate Dungeons

## Architecture Overview

This is a Next.js 16 application using multiple specialized data systems, each selected for its specific strengths within generous free tiers:

| System | Purpose | Status |
|--------|---------|--------|
| **Convex** | Real-time backend, schema, queries | Has full CRUD for games/characters/maps/items |
| **Neon PostgreSQL** | Game catalog (primary DB) | Used by `/api/games` REST routes |
| **MongoDB** | Extended game details | Characters, maps, items per game |
| **Redis** | Caching + job queues | 24h TTL cache + two queue processors |
| **Supabase** | Image storage | WebP conversion + upload |
| **Clerk** | Frontend auth + JWT provider | Active, wired to Convex and Supabase |

## Key Files

### API Routes

## JWT Integration

### How It Works
1. Clerk authenticates users and issues JWTs
2. `getToken({ template: 'serviceName' })` generates service-specific JWT
3. JWT is passed in `Authorization: Bearer <token>` header
4. Service validates JWT using shared secret
5. RLS policies can use `auth.jwt() ->> 'sub'` to get Clerk user ID

### Clerk JWT Templates

**supabase** (configured):
```json
{
  "app_metadata": {},
  "aud": "authenticated",
  "email": "{{user.primary_email_address}}",
  "role": "authenticated",
  "sub": "{{user.id}}",
  "user_metadata": {}
}
```
- Algorithm: HS256
- Signing key: Supabase JWT secret
