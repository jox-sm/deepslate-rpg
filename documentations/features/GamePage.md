Client-Side Trigger & Immediate NavigationAction: The user clicks on a game card component.Mechanism: Instead of making a blocking API request on the card itself, the app triggers an Immediate/Optimistic Navigation using the router.Route Format: It pushes the user directly to a dynamic route using the game's unique ID: /game/[uuid].UX Benefit: 0ms Latency UI. The game component mounts instantly with a loading skeleton while data is fetched in the background.2. Tiered Caching Lifecycle (Data Fetching Pipeline)When the game component mounts, it initiates a high-performance, multi-layered data retrieval pipeline:Tier 1: Client-Side Cache (First Line of Defense)The component checks the local client cache or state manager (e.g., Zustand, React Query) for the specific uuid.If Cache Hit: The game starts instantly.If Cache Miss: Proceed to Tier 2 via an API request.Tier 2: In-Memory Cache (Redis Lookup)The API handler queries Redis using the key game:{uuid}.Since Redis reads take roughly $1\text{ms} - 2\text{ms}$, this ensures ultra-fast delivery if the data was recently loaded.If Cache Miss: Proceed to Tier 3.Tier 3: Database Batch Fetching (MongoDB - Persistent Layer)If the data isn't in Redis, the request enters the Batch Accumulator.3. The Buffer/Flush Pattern (Smart Batch Loading)To shield MongoDB from handling hundreds of concurrent, isolated read requests, incoming dynamic data requests are buffered and grouped on the server.The Accumulator: Individual incoming requests for uuids are queued up in an in-memory buffer.The Flush Triggers: The server executes a bulk query to MongoDB based on a hybrid strategy ("Whichever Comes First"):Count-based Trigger: The buffer reaches a specific threshold (e.g., 100 requests).Time-based Trigger: A localized worker or interval triggers a flush every 1 second to ensure data doesn't sit stale if traffic is low.The DB Query: The worker pulls the batch list of uuids and executes a single, highly optimized bulk query using the $in operator:db.collection.find({ uuid: { $in: uuidBatchList } })* **Redis Repopulation:** Once MongoDB returns the batch results, the server writes these records into **Redis** with a designated Time-To-Live (TTL) so subsequent requests hit the memory layer instead.

### 🏆 Key Engineering Wins
* **Scalability:** Reduces database network roundtrips from $N$ individual requests down to a single batch trip, protecting connection pools.
* **Modular Design:** Keeps the logic clean, organized, and strictly optimized for read-heavy operations without cluttering code with spaghetti logic.
